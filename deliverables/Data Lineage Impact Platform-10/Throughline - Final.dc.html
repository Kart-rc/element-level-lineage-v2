How lineage is collected & scored
Confidence isn't a separate product — it's an attribute on every edge, shown inline wherever you inspect lineage. This page explains how that score is computed: a baseline extracted from code (static + LLM), kept current by a CI/CD diff on every change, and corroborated at runtime by Spark/Dask and OTel.
Why this tab Provenance is not a static explainer — it is the live trust layer behind everything else: every edge on the Lineage canvas carries a confidence score computed here (signals agreeing → verified; code-only → predicted), recomputed as each PR merges and each run reports. Low-confidence edges are drawn dashed amber on the Lineage canvas, so trust is visible where you read the flow — this tab is where you come to see why an edge scores what it does.

What a service comprises
A service node is a deployable unit, not just an API: its endpoints (REST/GraphQL), stream consumers & producers, and scheduled batch jobs. Each is a typed code path with its own channel — batch, API and streaming are all first-class, and expanding a service shows which unit reads or writes each dataset.
Looking for API calls?
Synchronous REST / GraphQL / gRPC and async events live in the Interactions lens — they move requests, not data, so they're kept off the lineage graph.

You never render thousands of nodes at once. Lineage is organised by altitude — start at domains, drill down only where you need to.

Baseline vs generated lineage
The baseline is what static + LLM predict from code. The generated lineage is what Spark/Dask/OTel observe the running service actually do. Reconciling the two is what produces the confidence score.
{{ r.glyph }} {{ r.title }} {{ r.tag }}
{{ r.body }}
Why reconcile, not replace
Static alone over-reports paths that never run; runtime alone misses rarely-fired or just-shipped paths. Keeping both — and scoring their agreement — is what makes impact analysis trustworthy.
