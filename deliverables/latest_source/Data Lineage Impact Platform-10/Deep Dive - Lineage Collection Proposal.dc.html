Lineage collection: the mechanism, its failure modes, and why it still wins

A feasibility-grounded proposal for the Throughline collection framework — every claim about third-party tooling cited to a primary source, every known failure mode stated before you find it.

Position

The two-plane architecture — a build-time predicted superset reconciled against runtime-observed evidence, with per-edge confidence — is the right design, and no off-the-shelf product delivers it. Two components of the original PRD are not feasible as specified (a Dask OpenLineage collector, which does not exist; and LLM self-reported confidence, which is empirically near-random) and are amended here. The single highest technical risk is not any collector — it is identity resolution across signals, and this proposal treats it accordingly: first, with numeric exit gates.

01 · Feasibility verdict

Component by component

ComponentVerdictGrounding
Static analysis, build time

Requires schema catalog access to resolve SELECT * and unqualified columns.

LLM self-reported confidence is near-random (AUROC 0.524) [3] — so it is removed from the scoring design. The Inferred-band cap + runtime-promotion gate stands as the containment mechanism.

Dask runtime Not feasible as specified "Dask hooks · OpenLineage" does not exist. Dask is not an OpenLineage integration [5]; no maintained lineage collector exists for it anywhere; Dask has no Catalyst-equivalent logical plan carrying column semantics. This is greenfield R&D — deferred and re-scoped, not silently rebranded as an integration task.
OTel as a lineage signal Legitimate only as scoped OpenTelemetry has no lineage semantic conventions, and the proposal to add lineage as an OTel signal was closed without adoption, deferring to OpenLineage [6]. Traces are trees over requests; lineage is a DAG over datasets [7]. OTel here confirms interaction edges and recency only — never column lineage.

Cross-cutting: the merge and confidence design is sound and partially executable already; identity resolution is named in PRD 1 as critical-path and has no design — §6 supplies one, with the Remediation Plan's numeric exit gates.

02 · Build-time plane

Tree-sitter → LLM, restructured as a three-tier ladder

The build-time plan is tree-sitter followed by LLM inference. That is defensible only if the LLM's decision surface is minimized. Tree-sitter produces a concrete syntax tree: no name resolution, no types, no data-flow. In a naive two-stage pipeline, every semantic judgment lands on the probabilistic component. The fix is to route each construct to the cheapest sufficient analyzer:

TIER 1
Deterministic parsers, wherever a grammar has real semantics
SQL → SQLGlot-class AST parsing (30+ dialects; benchmarked most accurate for column lineage [1]). dbt → compiled manifest. OpenAPI/protobuf specs → declared endpoints and schemas. Deterministic, reproducible, near-zero marginal cost. Never send well-formed SQL to an LLM — it costs more and is less accurate.
TIER 2
Tree-sitter as the candidate-finder and context-slicer
This is what tree-sitter is genuinely excellent at: 100+ grammars under one query mechanism, error-tolerant (parses half-broken code), incremental (fast per-PR re-parse). Structural queries deterministically locate the anchors — @KafkaListener("orders.events") annotations, S3 path literals, HTTP client calls, dataframe operations — and slice the exact enclosing function as context for tier 3.
TIER 3
LLM for the semantic residue only
Column mapping, transform expression, guard condition — the judgments with no deterministic path (dynamic SQL, UDF bodies, notebooks, stored procedures). Output is schema-validated JSON, cached by code hash, and quarantined in the Inferred band (score ≤ 64) until a runtime signal corroborates it. The promotion gate — not model confidence — is the safety mechanism.

Worked example: what each tier contributes

Source — OrderService.java (tree-sitter captures the underlined anchors deterministically)
// OrderService.java : 74–92
@KafkaListener(topics = "orders.events")          ← topic literal: tier 2
void onOrder(OrderEvent e) {
    if (e.getStatus() == CONFIRMED) {              ← guard: tier 3 names it
        row.put("total_amount", e.getAmount() / 100.0);  ← mapping + transform: tier 3
        writer.write("s3://lake/orders_raw/", row);  ← sink literal: tier 2
    }
}
Emitted candidate edge — schema-validated, provenance-tagged
{
  "src":  "urn:kafka:prod:orders.events#amount",
  "dst":  "urn:s3:lake/orders_raw#total_amount",
  "transform": "amount / 100.0",                    // tier 3 (LLM)
  "path": { "guard": "status == CONFIRMED",          // tier 3 (LLM)
            "codeRef": "OrderService.java:88" },     // tier 2 (tree-sitter)
  "signals": ["static", "llm"],
  "score": 48, "band": "Inferred"                    // capped until runtime agrees
}

Topic name, sink path, and code location are deterministic. Only the column mapping and transform are model-inferred — and they cannot influence a CI gate until production evidence confirms them (§5).

Evidence constraint we design around

Published LLM schema-lineage extraction: GPT-4.1 scores 0.418 zero-shot, 0.673 one-shot, 0.767 with chain-of-thought on the SLiCE metric; accuracy degrades as script complexity rises [2]. Self-reported confidence discriminates correct from incorrect output at AUROC 0.524 — a coin flip [3]. Consequences in this design: the LLM never scores itself, its edges never gate, and every extraction is cached by code hash so cost scales with churn, not estate size.

03 · The obvious challenge

"Why not CodeQL?"

CodeQL is the strongest static-analysis counter-proposal: it compiles code into a relational database with full name resolution, types, call graphs, and deterministic interprocedural data-flow / taint tracking — tracing e.getAmount() through helper methods into a sink is literally its core competency. The comparison, honestly:

Tree-sitter → LLM ladderCodeQL
Analysis depthSyntax tree + model-inferred semanticsSemantic DB: interprocedural data-flow, taint, types
DeterminismProbabilistic wherever tier 3 decidesFully deterministic
Language coverage100+ grammars, one mechanism~10 languages, one extractor each
Build requirementNone — parses raw source, tolerates broken codeCompiled languages need a buildable snapshot; extraction fails when the build fails
Per-PR latencySeconds, incrementalMinutes+ — DB extraction, then query evaluation
Dynamic SQLTier 3 infers intent from fragmentsTraces string construction, does not interpret SQL semantics
Notebooks, SQL files, DAG configsFirst-class inputsLargely outside the extraction model
LicensingOpen-source parsers + inference costPrivate commercial use requires GitHub Advanced Security licensing
Lineage fitEstate-wide candidate discoveryBespoke QL query packs per framework/idiom

Two conclusions. First, CodeQL is not a substitute: it fails the estate on language breadth, build requirements, artifact types, and licensing — and a lineage extractor in QL is a per-framework query-engineering project, not a switch to flip. Second, CodeQL is a credible later upgrade for precision on hot JVM paths, where its taint tracking could replace tier-3 inference with deterministic data-flow.

The decisive point — for either tool

Both approaches are repo-scoped. Neither can see that the topic orders.events written by repo A is the same physical dataset read by the Spark job in repo B. No static analyzer crosses the Kafka boundary, the S3 boundary, or the service boundary. That is not a tooling gap to fix at build time — it is the architectural argument for the runtime plane, and it is why we chose breadth (ladder) over depth (CodeQL) at build time and recover trust through runtime corroboration.

04 · Runtime plane

Each collector, with its failure modes attached

Spark — OpenLineage listener column-level

OpenLineageSparkListener registered via spark.extraListeners; column lineage on by default, extracted by traversing the Catalyst logical plan (Project, Aggregate, Join, Filter; DataSourceV2, Hive, JDBC relations) and emitted as the columnLineage facet [4]. The fault line is Catalyst vs everything else — not DataFrame vs SQL (both compile through Catalyst):

  • RDD operations — column extraction was deliberately disabled upstream because lost schema context produced incorrect lineage. RDD-heavy jobs yield dataset-level edges only.
  • Opaque UDFs — a UDF is a black box in the plan; columns entering it are traceable, the mapping inside is not. Tier-3 static extraction is the complement here.
  • JDBC relations — lineage recovered by parsing the SQL string: table/field names only, no types, no transform classification.
  • MLlib — no robust column lineage in any current collector. Treat as unsupported.
  • Facet omission is silent. By OpenLineage design, an integration that cannot compute column lineage simply omits the facet while still emitting dataset edges — so "we ingest OpenLineage" never implies column coverage. Coverage must be measured, not assumed (§10).

Warehouse-native federation column-level · zero instrumentation

For workloads already inside an engine, the engine's own lineage beats anything we could instrument:

  • Databricks Unity Catalog — automatic column-level lineage from Spark plans across all languages, queryable via system tables [8]. Caveat: "as much as possible" wording implies silent gaps; scope is Databricks-only.
  • Snowflake ACCESS_HISTORY — column-level reads and writes per query (objectsModified, columns); lineage is reconstructed from access records — excellent evidence, one derivation step removed from intent.
  • BigQuery / Dataplex — column lineage for some job types only; load jobs, routines, and external tables are uncovered. The most gap-ridden of the three; federate it, but publish its holes.

These are runtime evidence sources to federate, not to reimplement. They arrive as the same OpenLineage-shaped input to the merge engine.

Orchestrator & dbt job/dataset-level

  • Airflow — the OpenLineage provider is built into Airflow 2.7+ (AIP-53); extraction quality is per-operator, with a published supported-classes list [9]. Mostly dataset-level; treat column facets as operator-dependent bonus.
  • dbt — dbt-ol post-processes manifest.json / run_results.json after a run [10]. Artifact-based, not a live hook — which is fine: dbt's artifacts are the ground truth for its models, and tier-1 parsing of compiled dbt SQL supplies the column detail.

OTel traces interaction + recency only

Standard semconv spans give caller→callee (peer.service), endpoint (http.route, rpc.method), messaging destination (messaging.destination.name), volume, latency, and — critically for the confidence model — recency. What they do not and will not carry is column semantics: the OTel spec has no lineage conventions, and the proposal to add lineage as a signal was closed, deferring to OpenLineage [6]. Structurally, traces are per-request trees; lineage is a dataset DAG (an aggregation has no single parent request) [7]. Routing lineage through spans would be reinventing OpenLineage inside a data model that rejects it.

Dask — deferred, honestly

There is no OpenLineage Dask integration and no maintained Dask lineage collector anywhere [5]. Dask exposes a task graph (HighLevelGraph) — partition-level compute topology, not column lineage — and has no Catalyst-equivalent plan from which to extract column semantics; arbitrary Python UDFs make it strictly harder than Spark's UDF problem. Building this is legitimate R&D with publishable-novelty upside, and it is scheduled as exactly that: a Phase-3 decision driven by measured estate share, with tier-1/tier-3 static extraction covering Dask code meanwhile (its edges capped at Inferred, as designed).

The merge, as SQL over canonicalized edge tables
-- J1: column promotion
SELECT s.edge_id, 'spark' AS corroborating_signal
FROM   static_edges s
JOIN   ol_column_facets r
  ON   s.src_urn = canon(r.in_ns,  r.in_name,  r.in_field)
 AND   s.dst_urn = canon(r.out_ns, r.out_name, r.out_field);

-- J2 / J3: interaction + recency (note the catalog hop)
SELECT s.edge_id, o.last_seen, o.count_24h
FROM   static_call_edges s
JOIN   svc_catalog c  ON s.repo = c.repo            -- repo → service.name
JOIN   otel_agg o     ON c.otel_service = o.service_name
                     AND s.declared_route = o.http_route;

06 · Identity resolution

The linchpin, treated as such

Everything downstream — dedup, attribute union, scoring, gating — presupposes that five signals agree on what "the same thing" is. The design, deterministic-first:

  • Canonical URN scheme — urn:<system>:<instance>:<path>#<element>, with per-system normalizers (S3 path canonicalization, topic env-prefix stripping rules, warehouse three-part naming, route templating enforcement). Normalizers are code-reviewed config, not inference.
  • Service catalog as the identity backbone — the repo ↔ service.name ↔ owner mapping is maintained where it already lives (catalog/CODEOWNERS), and joins go through it; OTel service.name conformance becomes an onboarding requirement, checked in CI.
  • Deterministic match first, fuzzy fallback quarantined — exact-URN matches auto-merge; fuzzy candidates (edit distance, schema-shape similarity) go to a review queue and never silently merge.
  • Measured, with exit gates — PI-1 spike on the vertical slice against hand-labeled ground truth: GO ≥ 95% precision and ≥ 90% recall; PIVOT = add a manual mapping layer; STOP = re-scope to per-system lineage with no cross-signal merge. The STOP branch is stated because it is real: without reliable identity, the confidence model that Impact gating depends on does not exist.
07 · Confidence engine

The math, its strengths, and what we changed

Scoring — as specified in PRD 1 (executable today)
weights   = { static: 30, llm: 18, spark: 30, dask: 26, otel: 22 }
recency   = { fresh_1d: 1.0, aging_7d: 0.82, never: 0 }   // runtime terms only
score     = min(100, baseline + Σ runtime_i × recency_i)
if (no runtime signal) score = min(score, 64)             // the cap
band      = score ≥ 85 && anyRuntime ? Verified : score ≥ 65 ? Probable : Inferred

What is right about it: the baseline-only cap at 64 means no purely static/LLM edge can reach gating territory — this single constraint contains the LLM's failure modes better than any accuracy improvement could. Recency decay on runtime terms only is correct: code doesn't get staler, observations do. Full per-edge provenance (signals[]) makes every score auditable.

What we changed and added:

  • LLM self-reported confidence removed from the formula's inputs (it was specified in FR-C2 but never wired in — correctly, as it turns out: AUROC 0.524 [3]). External signals replace it: tier-1 parser cross-check, self-consistency across samples, and runtime corroboration.
  • Calibration plan — weights are self-described as heuristic. The vertical slice produces labeled outcomes (edges later confirmed vs retired); weights get fit to those outcomes per signal per stack, and re-fit on a schedule. A confidence score that is never calibrated against reality is a UI decoration.
  • Conflict-resolution matrix — attribute union assumes non-conflicting attributes; conflicts need a policy, not an accident of ingestion order:
ConflictResolution
Static asserts edge; runtime never observesEdge stays Inferred; cold-path decay applies; Curator question after N windows
Two signals report different transforms for one edgeRuntime facet wins for what happened; static wins for expression text; disagreement itself lowers score and is surfaced
Runtime observes edge static never predictedEdge admitted as runtime-only, flagged drift; counts against extractor recall
Two collectors disagree on schema/typeRegistry (schema registry / warehouse catalog) is authoritative; collectors annotate

09 · The argument

Why the hybrid still wins

Every alternative above fails the same test: can it tell you, in the pull request, what a change breaks — including on paths that haven't run lately — and how much to trust that answer?

  • Completeness requires build time. Only code analysis sees the superset — every branch, guard, and feature-flagged path, including the ones that will fire next quarter. Runtime can never testify about a path that hasn't executed.
  • Trust requires runtime. Only observation separates live edges from dead code and wrong inferences. Static analysis can never prove its own assertions.
  • Gating requires the confidence contract. A gate that fires on unproven edges gets disabled by the second false positive. The band system — with prediction hard-capped below Verified — is what makes automation socially survivable.
  • The seams are where the work is. Identity resolution and merge policy are the actual hard problems, which is exactly why they are sequenced first with numeric exit gates, rather than discovered in month nine.

The composition — predicted superset, observed confirmation, calibrated confidence, per-PR refresh — is not available to buy, and each pillar uses the most boring proven technology available for its job: SQLGlot-class parsing, the OpenLineage ecosystem, warehouse system tables, standard OTel semconv. The novelty budget is spent in exactly two places: the reconciliation layer and the tier-3 residue. Everything else is deliberately unoriginal.

10 · Open problems

Stated plainly, before you find them

ProblemStatus
Cold and seasonal pathsA never-observed path stays Inferred no matter how critical — "low confidence ≠ low risk." Mitigations (Curator owner-questions, branch-coverage reporting, decay windows sized per path class) reduce but do not eliminate this. Honest answer: the tail is verified by humans, prompted by the system.
"Element-level" beyond columnsOpenLineage's facet is column→column; intra-job transformation chains and sub-column semantics are unstandardized (open proposal, OL #4090 [11]). Our transform + path attributes carry finer detail than the standard — a deliberate extension, flagged as such.
LLM cost & model driftCode-hash caching bounds cost by churn; extraction is versioned by model+prompt so a model upgrade re-scores rather than silently rewrites. A standing eval set (labeled slice) is the regression suite. Budget is a published number per PI, not an assumption.
Coverage tailFlink, Kafka Streams, stored procedures, legacy ETL, BI tools have no collector yet. The coverage map (per system: collector, granularity, confidence ceiling) is a Phase-1 deliverable, and the UI must render "not yet observed" distinctly from "no dependency."
Security & PII postureSource code and SQL flow to an LLM; column classifications flow through the graph. Requires: scoped model endpoints, no training on submitted code, secret-scanning before prompt assembly, and classification-aware access control on the graph itself. Unwritten in PRD 1; must be written before Phase 1.
Graph store selectionSizing (~1M nodes, ~3M edges, ~500K events/day at 10K services) is order-of-magnitude, unvalidated against a concrete engine. Phase-0 slice doubles as the storage bake-off.
