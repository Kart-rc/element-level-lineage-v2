Architecture Brief v1.0 · For technology leadership Owner: Data Platform

Lineage collection, at a glance

One canonical, column-level lineage graph built from two independent kinds of evidence — what the code says, and what production proves — with a confidence score on every edge.

The one-paragraph version

We parse every repo to predict the complete set of data flows — every branch, including paths that rarely run. We then observe production — Spark runs, warehouse query logs, orchestrator runs, service traces — to confirm which of those flows are real and current. Where prediction and observation agree, an edge is Verified and safe to gate a code change on. Where only the code asserts it, the edge is visibly Inferred and never blocks anyone. That honesty — a confidence score on every edge — is what makes the graph trustworthy enough to put in front of a merge button.

01 · The problem

A schema change is a bet placed in the dark

Nobody can answer "if I drop this column, what breaks?" before the change ships. Hand-curated lineage rots the week after it's drawn. Catalog products stop at table-level lineage inside their own connectors. And any single automated source is incomplete: code analysis sees paths that never run; runtime observation misses paths that haven't run yet — the quarter-end job, the error branch, the EU-only flow.

The wedge is not "another catalog." It is column-level impact answers, delivered in the pull request, with an explicit statement of how much to trust each answer. No off-the-shelf product does that today.

02 · The concept

Two planes of evidence, reconciled

Every collection mechanism we run belongs to one of two planes:

PlaneWhat it knowsIts blind spot
Build time · predicted The complete superset of flows the code can produce — every branch, guard, and transform, tied to a file and line. Unproven. Includes dead code; a parser or model can be wrong.
Runtime · observed What actually executed — real reads and writes, real column derivations, real call volumes, with timestamps. Only sees paths that fired. Silent about the rare-but-critical tail.

Neither plane is sufficient alone. Reconciled, they cover each other's blind spot: build time supplies completeness, runtime supplies proof, and the confidence score is the measure of their agreement and its recency. Every code change re-runs the loop: predict in CI, watch after deploy, confirm in production.

05 · Trust

Three bands, one rule

≥ 85
Verified
Code and runtime agree, recently. The only band allowed to block a merge.
65–84
Probable
Strong evidence, imperfect agreement or aging observation. Warns, never blocks.
< 65
Inferred
Asserted by code analysis alone. Drawn dashed; excluded from automated gates.

The one rule: prediction alone is capped below Verified. An edge that only the code asserts — however plausible — can never score above 64, so it can never gate a change. Only production evidence promotes an edge into gating territory. This single constraint is what keeps LLM-assisted extraction safe to use at all.

06 · De-risking

Four amendments to the original plan

A feasibility review of PRD 1 against the current ecosystem (primary sources: OpenLineage & OpenTelemetry specs, DataHub's parser benchmark, published LLM-extraction studies) confirmed the architecture and corrected four specifics:

ChangeWhy
Parser-first, LLM-residual Deterministic SQL parsers are benchmarked as the most accurate source of column lineage. The LLM handles only what parsers can't — dynamic SQL, UDFs, notebooks — cached by code hash. Cheaper, more accurate, reproducible.
Dask deferred; warehouse-native + orchestrator collectors added No Dask lineage integration exists anywhere — it is greenfield R&D, not configuration. Warehouse-native lineage and the Airflow/dbt collectors deliver strictly more estate coverage in the same phase, at connector-level effort.
OTel scoped to interactions OpenTelemetry formally declined data lineage as a signal; OpenLineage is the standard. OTel remains exactly what it is good at: the live service-interaction map and a recency pulse on every flow.
OpenLineage end-to-end Every collector emits one standard event format. We differentiate on collection breadth, confidence, and CI gating — not on reinventing metadata plumbing an ecosystem already maintains.

08 · Risks

The three that matter

Risk 01
Identity resolution
The same dataset is named differently by every signal. A missed match doesn't error — the graph silently under-reports.
Mitigation: P0 spike with numeric exit gates; canonical URN scheme; deterministic matching first, catalog-backed.
Risk 02
Coverage tail
Systems without a collector become invisible edges while the UI implies completeness.
Mitigation: published coverage map; explicit "not yet observed" state in the UI; federation strategy prioritized by estate share.
Risk 03
LLM accuracy & cost
Published extraction accuracy tops out ~0.77 on hard cases, and models are systematically overconfident.
Mitigation: parsers first; LLM residual-only, cached by code hash; hard cap below Verified until runtime corroborates.
