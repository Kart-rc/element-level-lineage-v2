Throughline
End-to-end data lineage capture & impact analysis — design directions
Six directions to react to: three for the lineage canvas, three for impact analysis. All run on one universal graph spanning services, external sources, a stream platform, S3 landing, analytical apps, S3 files and file transfers — with many-to-many, multi-pattern flows: data may originate from a service, land directly from a 3rd-party source, skip the stream, or flow back into an operational system.
System type · color
Service Stream topic S3 landing Analytical S3 file File transfer External src
Topology model
Each node is one of four roles — shape encodes it, the badge labels it. Columns are flow stage (hops from a source), not system type.
Service  ● CIRCLE
Our app that originates or receives data — incl. reverse-ETL targets.
Source  ○ RING
External 3rd-party feed — lands files or pulls APIs straight into S3.
Job  ◇ DIAMOND
A process that reads & writes datasets — dbt / Spark / SFTP runs.
Dataset  ▢ SQUARE
Data at rest with a schema — topics, S3 tables, files.
Edges & flow patterns
Inferred from each job run's input / output facets and publish bindings — never hand-drawn.
Flows are not fixed: external feeds skip services & stream, services can export straight to S3, and outputs can flow back to a service.
