Impact Analysis

Given a proposed or detected change to any upstream element, Throughline computes the blast radius across the lineage graph — at service, dataset, and column level — and classifies each consumer by how badly it breaks.

Executive summary

A schema change in one service can silently break a partner export three hops away. Today that’s discovered in production. Throughline turns it into a pre-merge question: select a change, see exactly which applications, datasets, columns, and downstream contracts are affected — ranked breaking › warning › safe — and who owns each. The same engine runs three ways: an interactive blast-radius on the graph, a simulation report before you commit, and an automatic CI/CD gate on every PR.

Three modes, one engine Confidence-aware severity Owner-routed alerts

01Goals & non-goals

Goals
  • ✓Answer “what breaks if I change X?” at column granularity in seconds.
  • ✓Classify every affected consumer by severity and route to its owner.
  • ✓Run before merge (simulation) and on merge (CI/CD gate).
  • ✓Reflect confidence — never block on an unverified inferred edge.
Non-goals (v1)
  • —Auto-generating the fix or migration code.
  • —Predicting data-quality / statistical drift (separate concern).
  • —Cost/performance impact estimation in v1.

05Confidence-aware severity

Impact rides on lineage edges, and edges carry confidence (PRD 1). The blast radius is therefore reported in two tiers so an inferred edge never silently blocks a deploy.

Confirmed impact

Reached only via Verified / Probable edges. Counts toward the CI/CD gate; can block a merge.

Possible impact

Reached via at least one Inferred edge. Surfaced for review, shown dashed, never auto-blocks.

06Code-path coverage

A change to a column affects every code path that reads it — not just the common one. Impact starts from the static superset of paths (PRD 1) and tiers each by how often it actually executes. The execution class maps directly onto Confirmed vs Possible impact.
