Lineage Collection

How Throughline builds and continuously trusts a single, organization-wide lineage graph from code, runtime, and change events — with a confidence score on every edge.

Executive summary

Lineage is only useful if people trust it. Hand-curated lineage rots; single-source automated lineage is incomplete. Throughline collects lineage from four independent signals — static code analysis with LLM inference, a CI/CD diff on every change, runtime collectors (Spark & Dask), and OpenTelemetry traces — then reconciles them into one canonical graph where every edge carries a confidence score derived from the signals that agree.

Scales to 2,000+ services & datasets Batch and streaming Prove value on one vertical slice first

01Goals & non-goals

Goals
  • ✓One canonical, column-level lineage graph spanning services, streams, S3 and file transfers.
  • ✓A calibrated confidence score on every node and edge, with full provenance.
  • ✓Lineage kept current automatically as code changes — no manual upkeep.
  • ✓Collectors that degrade gracefully when a signal is missing.
Non-goals (v1)
  • —Auto-remediation of broken pipelines.
  • —A data catalog / glossary product (we integrate, not replace).
  • —Real-time (sub-second) lineage; near-real-time is sufficient.
  • —Every language/framework at launch — start with the vertical slice's stack.

Path-qualified edges why a service has many edges

A service is not one transform. The same input column reaches different outputs depending on the code path taken — branches, event-type switches, feature flags, null/error handling. Static + LLM analysis emits the superset of all potential paths; runtime collectors confirm which actually execute. Each edge therefore carries the path that produced it:

{{ p.key }} {{ p.desc }}
Superset · static + LLM

Every branch, including rarely-taken and error paths. Complete, but over-approximates — may include dead code.

Executed subset · runtime

Which paths actually fired, and how often (hot/warm/cold). Confirms reality, but a never-yet-observed path stays unconfirmed.

Per-path execution frequency feeds confidence: a hot, runtime-confirmed path scores Verified; a static-only branch stays Inferred. Branch coverage — the share of code paths runtime-confirmed — becomes a measurable risk surface (see PRD 2 §4).

What each signal collects — and what it powers

No single collector sees everything. Static + LLM proposes the complete shape; the runtime signals confirm reality and add frequency, freshness and traffic. The merge engine unions their attributes onto one entity/edge record.
